# extract-approach — 解法レコーダー

## 目的

セッションが終わると消える「上位モデルの思考プロセス」をリポジトリ内に固定する。
記録されたノートは、後続のどのモデル(Opus/Sonnet/Codex)でも読んで適用できる。
答えは1回きりだが、書き残されたアプローチは以後の全セッションを底上げする。

## トリガー(該当したら発火)

- 最初のアプローチが失敗し、方針転換して解決した(opus-task-loopのループが転換を経て完了した場合は典型例)
- 複数の設計候補を比較して1つに決めた(トレードオフ判断があった)
- 原因が非自明だったバグの根本原因に到達した
- 「なぜそうするか」を説明しないと次の人が同じ穴に落ちる、と感じた
- Fable等の上位モデルで判断を確定した(fable-escalation経由) — このセットの存在理由そのもの。上位モデルの判断こそ最優先で記録する

**該当しない場合**: 1回で素直に解けた問題、既知パターンの適用、機械的作業 → 記録しない(ノイズを増やさない)。

## 出力

1解決 = 1ノート。**作成前に**出力先ディレクトリを検索し、同テーマの既存ノートがあれば新規作成せず追記・更新する。場所:

- gitリポジトリ内で作業中: `<repo>/docs/learnings/YYYY-MM-DD-<slug>.md`
- リポジトリ外: `~/tasks/learnings/YYYY-MM-DD-<slug>.md`

`<slug>` は問題を表す短いkebab-case(例: `sqlite-wal-lock-contention`)。

## ノート形式

```markdown
# <問題の一行要約>

- 日付: YYYY-MM-DD
- 領域: <repo名/サブシステム>
- 種別: bug | architecture | debugging | design-tradeoff

## 問題
何が起きていたか / 何を決める必要があったか。再現条件や制約を含める。

## 試して駄目だった道
各アプローチと、なぜ捨てたか。1行ずつでよい。ここが最も価値が高い —
次のモデルが同じ袋小路に入るのを防ぐ。

## 効いたアプローチ
実際に解決した手順・判断。コードは最小限の抜粋のみ。

## なぜ効いたか
根本原因、または判断の決め手。「動いたから」ではなく機構を書く。
裏づけ(実行したテスト・観測したログ・差分など、独立検証の証拠)を1行添える。

## 一般化できる原則
この解決から抽出できる、次の類似問題に適用可能なルール。1〜3行。
チェック可能な形(「〜の場合は〜であること」)で書く。
無理に一般化しない — 固有すぎる場合は「なし(この文脈固有)」と明記してよい。
```

## 書き方の基準(合否判定)

- [ ] 「試して駄目だった道」に、失敗した試行**または比較検討して却下した候補**が最低1つ書かれている(どちらも無ければこのスキルの対象ではない可能性が高い)
- [ ] 「なぜ効いたか」が機構で説明されている(「再起動したら直った」は不合格)
- [ ] 「一般化できる原則」が、このリポジトリ固有の文脈なしでも読める
- [ ] ノート全体が1画面〜2画面(長い=蒸留されていない)

## 既存の仕組みとの役割分担

- `~/tasks/lessons.md` — **ワークフロー/プロセスの教訓**(ツールの使い方、ユーザーとの作業ルール)。従来どおりそちらへ。
- auto-memory — ユーザー・環境に関する**事実**。従来どおりそちらへ。
- このスキル — **技術的な解法と判断の記録**。コードベースに紐づくものはリポジトリに置き、コードと一緒にバージョン管理する。

迷ったら: 「次に同じ*技術的*問題に当たったモデルに読ませたいか?」→ Yes ならこのスキル。

## ノート作成後

- ノートのパスをユーザーへの完了報告に1行で含める(内容の再説明は不要)

## 読む側の配線(recall)

書くだけでは資産にならない。深いデバッグ・設計判断・非自明な調査を**始める前に**、
対象リポジトリの `docs/learnings/`(リポジトリ外なら `~/tasks/learnings/`)を
テーマで検索し、既存ノートがあれば先に読む。

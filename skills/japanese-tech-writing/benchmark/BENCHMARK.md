# ベンチマーク：japanese-tech-writing スキルのあり／なし

対象タスク：[interpret-paper](https://github.com/hikimay/interpret-paper) の `README.md` を、リポジトリの目的・要件・使い方という同一の入力情報から生成する。

- 条件 A（スキルなし）：[README_no_skill.md](README_no_skill.md) — スキルを読み込まず、生成 AI が既定で出しがちな日本語の register を模して生成。
- 条件 B（スキルあり）：[README_with_skill.md](README_with_skill.md) — [rules.md](../reference/rules.md) と [ng-vocab.md](../reference/ng-vocab.md) を適用して生成。

両者の事実内容（コマンド、取得順、ディレクトリ構成など）はそろえ、文章の規範だけを変数とした。

## 採点結果

[checklist.md](../reference/checklist.md) 末尾の 6 軸ルーブリック（各 10 点、合計 42/60 で合格、0〜2 点の軸が一つでもあれば書き直し）で採点した。

| 軸 | A（なし） | B（あり） | 主な差 |
|---|---:|---:|---|
| 主張 | 3 | 8 | A は「非常に強力」等の一般論。B は対象読者と機能を具体的に述べる |
| 厳密さ | 3 | 9 | A の「賢く使い分けて／最適なコンテンツを自動取得」は実装（固定順）と食い違う誇張。B は取得順を正確に記述 |
| 主体性 | 3 | 8 | A は行為者を「AI／ツール」で曖昧化。B は何が何をするかを名指し |
| 具体性 | 3 | 9 | A は `--detail` の水準・取得順・対象読者を省く。B はすべて明記 |
| 簡潔さ | 3 | 9 | A は「とても簡単です。〜だけで、すぐに使い始められます」等の中間説明が冗長 |
| 語り | 2 | 9 | A は絵文字・予告総括・決め台詞・強度副詞が多い。B は演出を排除 |
| **合計** | **17/60** | **52/60** | A は不合格かつ語り軸 ≤2 で書き直し対象。B は合格 |

## 所見

スキルなしの README は、体裁は整っていても「賢く使い分けて最適なコンテンツを自動取得」のように実装と食い違う誇張を含み、`--detail` の水準や取得順といった利用者が必要とする具体情報を雰囲気の文に置き換えていた。スキルありでは演出語を削った結果、同じ分量の枠内で具体情報が増え、厳密さ（観点 3）と具体性（観点 4）の点が上がった。

このベンチマークで効いたのは、語り・簡潔さの見た目の改善より、観点 3 と 4 である。演出を外すと埋め草の文が減り、その分を事実で埋める必要が生じるため、記述が正確かつ具体的になった。

## 方法上の限界

- 条件 A・B は同一のモデルが生成しており、独立した二つの実行ではない。A はモデルが既定で出しやすい register を模したものであって、意図的に劣化させた strawman ではないが、人手の盲検生成ではない。
- 採点は同一スキルのルーブリックを自己適用したもので、第三者による盲検評価ではない。点数は相対差の目安として読む。
- 参考までに、interpret-paper の実リポジトリの README は本スキルなしでも整っており（半角スペースやコードブロックの扱いが適切）、条件 A のような既定 register の文章とは異なる。本ベンチマークの A は「素の生成で起こりやすい劣化」を示すものである。

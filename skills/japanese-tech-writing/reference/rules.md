# 日本語技術文書の文章規範（詳細）

出典：k16shikano「日本語技術文書の文章規範」
<https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d>

技術文書を生成・推敲する際に守る規範を、観点ごとにまとめる。各項目は「何を守るか」と、可能なら「NG → 望ましい書き換え」を示す。

---

## 1. 整形

- 一文ごとに改行する。段落は空行で区切る。
- 日本語の中に半角の英数字（英単語、製品名、コード、数値＋半角単位など）が入るときは、その前後に半角スペースを入れる。
  - NG：「これはLLMの特徴である」「Pythonで書く」 → OK：「これは LLM の特徴である」「Python で書く」
  - 約物（句読点、全角括弧など）に隣接する場合はスペースを重ねない。NG：「（ API ）」 → OK：「（API）」
  - 全角文字どうしの間や、英数字が文頭・文末にある場合は不要。
- コード、ログ、コマンド、出力はコードブロックに入れる。本文に混ぜない。
- 本筋から外れる補足は脚注に降ろす。本文の論理を途切れさせない。
- 用語の初出は**太字**で示し、以後は「」で表記する。
- ダッシュ（—）や中黒（・）の使用を控える。区切りは句読点で済ませる。
- 見出しは単一のトピックに絞る。複数の話題を一つの見出しに詰めない。
- 用語定義は全角コロンを使い「**用語**：説明」の形式で書く。

## 2. 段落と論証の構成

- 一段落につき一トピック。段落内で話題を切り替えない。
- 段落の冒頭で、その段落が何を述べるかを明示する。
- 接続表現で、前の段落との論理関係（だから／一方／たとえば等）を示す。
- 論証は一方向に進める。行きつ戻りつしない。結論は一度だけ述べる。
- 読者が誤って解釈しそうな点は、その誤りを**明示的に否定してから**正しい説明をする。
- 山場の重要情報を、効果を狙って先に出してしまわない。論理の順序で出す。

## 3. 論証の厳密さ（衝突時の最優先）

- 「かもしれない」などの推量表現を、根拠なく削らない。不確実性が実在するなら残す。逆に、根拠なく断定するために推量を消すのも禁止。
- 異なるものを、似ているからといって同じものとしてまとめない。
- 複数の要因がある事象を、単一の原因に還元しない。
- 章や節をまたいで、概念の定義を一致させる。同じ語を別の意味で使わない。
- 因果関係は「AだからB」で止めず、**どういう機構でそうなるか**まで説明する。
- 「必ず」「絶対に」「常に」のような絶対的な言い方を避ける。例外の余地を残す。

## 4. 読み手の負荷の管理

- 論証に不要な固有名詞（製品名、人名、ライブラリ名など）を出さない。
- 抽象的な表現には、括弧で具体例や言い換えを補う。
- 新しい例を出すときは、なぜその例が必要かを前置きで説明してから出す。
- 冒頭で詳細を盛り込みすぎない。まず全体像、詳細は後。

## 5. 視点と語り

- 受動態でぼかさず、能動態で行為者を主語にする。
  - NG：「設定が変更される」 → OK：「管理者が設定を変更する」
- 読者を「あなた」と呼ばず、役割名（開発者、利用者、管理者など）で表記する。
- 「AI」「ツール」「システム」のような曖昧語を避け、何を指すか具体的に書く。
- 一度導入した術語で最後まで通す。同じ概念を別の語で言い換えない。
- 訳語は、その分野で慣用されているものを選ぶ。独自の訳をあてない。

## 6. 演出の抑制

- 説明で足りる箇所に、溜め・引き・修辞疑問を使わない。
- 短い決め台詞を独立した段落に置いて緊張を作る、といった演出をしない。
- 太字での強調は、**誤読の防止**と**結論・帰結の明示**に限る。雰囲気作りに使わない。
- 命令調（「〜せよ」「〜すべきだ」）より、「〜するわけにはいかない」のような事実に即した言い方を選ぶ。
- 帰結を列挙して事故や危険を煽らない。転回点を過剰に劇的にしない。
- 慣用句をひねった言い回しや、解釈が一意に決まらない比喩を使わない。

## 7. LLMっぽい表現の禁止

中身がないのに「書いている感」だけを与える型を排除する。

### 予告と総括の型

- NG：「重要なのは〜である」「ここで押さえておきたいのは〜だ」「多角的に分析すると〜」
- OK：予告や総括を挟まず、その内容を直接書く。
  - NG：「多角的に分析すると、重要なのは、正しさを誰が知っているかである。」
  - OK：「評価の核心は、正しさを誰が知っているかにある。」

### 「正面から」系（姿勢だけの宣言）

- NG：「本章では、〇〇の理論を**正面から**扱う」「この前提を、ここで**正面から**回収する」
- OK：「本章では、〇〇の理論を扱う」「ここで、この前提を回収する」

### 空虚な形容詞・動詞

- 避ける語の例：「不可欠」「核心的」「掘り下げる」「言及する」「〜において」「〜という観点から」
- これらは具体的な動作・対象に置き換えるか、削る。
  - NG：「この概念について掘り下げる」 → OK：「この概念がどの場合に成立するかを示す」

## 8. 冗長の排除

- 同じ主張を語句を変えて繰り返さない。一度で言い切る。
- 場面や手順を描写した直後に、その要約をもう一度書かない。
- 並列する事実は一文にまとめる。一事実一文に分けて水増ししない。
- 読者が前後から補える中間説明を書かない。
- 接続や評価のためだけの文（「このように」だけの文など）を置かない。
- 想像上の読者との問答（「ここで疑問が湧くかもしれない。それは〜」）を入れない。
- メタな枠取りを避ける。
  - NG：「これは〇〇という発想である」「これが〇〇の考え方だ」 → OK：その発想・考え方の中身を直接書く。

## 衝突したときの優先順位

1. **論証の厳密さ（観点3）** — 正確さと誠実さ。必要な不確実性は残す。
2. **読み手の負荷の管理（観点4）・冗長の排除（観点8）** — 分かりやすさと簡潔さ。
3. その他の形式・演出に関する観点。

簡潔にするために事実を曲げたり、必要な留保を消すことは、いかなる場合も認めない。

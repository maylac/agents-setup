# 並列監査サブエージェントの所見矛盾は「安い直接再検証」で決着させてから設計に載せる

- 日付: 2026-07-17
- 領域: agents-setup / クロスハーネス監査(claude/codex/hermes)
- 種別: debugging + design-tradeoff

## 問題
3並列Explore監査の所見を基にOpus設計エージェントが是正案を作ったところ、載っている「事実」が2件相互矛盾した。
1. Orca残骸: Explore#1「configが本日再生成・app現存」 vs 設計側「app/config/プロセス全て不在」
2. Codex設定意図: Explore#3「文書hooks.mdの意図はnever/danger-full-access(実設定が乖離)」 vs 設計側「state DBにユーザーの『danger-full-accessはだめでしょ』発言があり、現設定こそ意図」

矛盾を放置してどちらかを信じると、①存在しないOrcaの掃除タスクを実行、②ユーザーが明示拒否した設定への「回帰」を是正として実行、という2つの誤爆になるところだった(実際、②はAskUserQuestionの選択肢説明に誤前提のまま混入し、ユーザーが承認してしまっていた)。

## 試して駄目だった道
- 後発エージェント(設計側)を新しい方として自動的に信じる — Orca件は後発が正しかったが、その判定基準では②を外す(どちらが正しいかは新旧でなく検証手段の質で決まる)
- 多数決/再度サブエージェントに調査させる — 追加コンテキスト消費の割に、同じ盲点(ユーザーレベル~/Applicationsと/Applicationsの見る場所違い等)を再生産するリスクが残る

## 効いたアプローチ
矛盾した所見**だけ**を親セッションが直接1コマンドずつ再検証した(計2バッチ、数秒)。
- Orca: `ls -d ~/Applications/Orca.app` + Application Supportパス + `pgrep` → 全て不在で決着
- hooks.md: `grep -rn danger-full-access claude/rules/` → ファイルは実在、ただし内容はstale(拒否発言の方が新しい)と確定
その上で、誤前提で得ていたユーザー承認を**訂正付きで再提示**し(danger-full-accessへは変更せず never+workspace-write の中間解)、プラン本文に訂正経緯を明記した。

## なぜ効いたか
サブエージェントの所見は「観測時点+観測方法」に依存し、報告文には方法の盲点が現れない(#1はユーザーレベルApplicationsを見て、設計側は/Applicationsを見ていた等)。矛盾は「どちらかが嘘」ではなく「観測方法が違う」シグナルなので、方法を固定した最小の直接観測が最も安い決着手段になる。裏づけ: 2件とも1バッチのBashで確定し、是正計画から誤タスク2系統(Orca掃除/危険側回帰)を除去できた。

## 一般化できる原則
- 複数サブエージェントの所見が矛盾したら、多数決や再委譲でなく「矛盾点だけを親が直接再検証」してから下流(設計・承認・実行)に流すこと
- サブエージェント所見を根拠にユーザー承認を取った後で前提が覆った場合、承認を有効として実行せず、訂正付きで再提示すること
- 「意図」を文書から読む場合、実設定・state DB・バックアップ世代の時系列と突き合わせ、最新のユーザー発言を優先すること
